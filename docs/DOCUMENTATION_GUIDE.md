# 编码文档规范

进行 Vibe Coding 时，文档的核心作用不是“写得完整”，而是让 agent 随时知道：

1. 项目要解决什么问题
2. 当前代码为什么这样设计
3. 现在做到哪里了
4. 接下来应该做什么
5. 什么情况下才算完成

相比传统开发，Vibe Coding 更需要文档成为项目的“长期记忆”。建议按下面几类维护。

## 一、项目定位与需求文档

### 1. `README.md`

项目入口文档，负责帮助人和 Agent 快速建立整体认知。

建议包含：

* 项目简介
* 核心功能
* 技术栈
* 项目目录说明
* 本地启动方式
* 环境变量说明
* 测试和构建命令
* 其他文档入口

README 不适合承载大量需求和设计细节，它更像项目地图。

### 2. `PRODUCT.md` 或 `PRD.md`

描述“做什么”和“为什么做”。

建议包含：

* 产品背景
* 目标用户
* 核心使用场景
* 功能范围
* 非功能需求
* MVP 边界
* 明确不做的功能
* 验收标准

对于 Agent，尤其要明确：

```md
## 本阶段不实现

- 不实现第三方登录
- 不实现移动端
- 不实现多租户
- 不提前设计插件系统
```

“不做什么”往往比“做什么”更重要，可以防止 Agent 过度设计。

---

## 二、项目计划与进度追踪文档

### 3. `ROADMAP.md`

描述项目的阶段规划，而不是具体编码任务。

例如：

```md
## S0：基础可用版本

- 用户登录
- 简历上传
- AI 面试
- 面试报告

## S1：面试能力增强

- 动态追问
- 压力面试
- 面试官风格

## S2：职业发展

- 职业测评
- 职业规划
- 简历优化
```

每个阶段建议标记：

* 目标
* 范围
* 前置依赖
* 完成条件
* 当前状态

### 4. `TASKS.md` 或 Issue Backlog

用于记录具体任务，是 Agent 每次开始工作前最应该阅读的文档之一。

推荐格式：

```md
## 正在进行

### TASK-023 实现简历解析接口

状态：进行中

目标：
上传 PDF 简历并返回结构化数据。

验收标准：
- 支持 PDF 文件
- 文件最大 10 MB
- 返回姓名、教育经历、工作经历
- 解析失败时返回明确错误
- 包含单元测试

相关文件：
- src/main/java/.../ResumeController.java
- docs/api/resume.md

依赖：
- TASK-018 文件存储模块

当前进展：
- 上传接口已完成
- PDF 文本提取已完成
- 尚未实现结构化解析
```

比起简单写一句“实现简历解析”，这种任务描述更适合 Agent 执行。

### 5. `STATUS.md`

记录项目当前真实状态，解决“文档规划”和“代码实际情况”不一致的问题。

建议包含：

```md
## 当前版本

S0 开发阶段，完成度约 65%。

## 已完成

- 用户登录
- 简历上传
- 简历文本提取

## 正在进行

- 候选人画像生成

## 尚未开始

- AI 面试
- 面试报告

## 当前阻塞

- 大模型结构化输出偶尔不稳定

## 下一步

1. 完成画像生成的 JSON Schema
2. 增加异常重试
3. 编写集成测试
```

Agent 每次结束较大的开发任务后，应同步更新这个文件。

---

## 三、架构与技术设计文档

### 6. `ARCHITECTURE.md`

描述系统整体结构，以及各模块之间如何协作。

建议包含：

* 系统架构图
* 模块职责
* 数据流
* 调用关系
* 分层规则
* 模块边界
* 外部依赖
* 部署结构
* 关键约束

例如：

```md
## 分层约束

Controller：
- 只负责参数校验和响应转换
- 不允许直接访问 Repository

Service：
- 负责业务逻辑
- 事务在此层管理

Repository：
- 仅负责数据访问
- 不包含业务判断
```

这种明确约束可以显著减少 Agent 生成不符合项目架构的代码。

### 7. `ADR/` 架构决策记录

ADR，即 Architecture Decision Record，用来记录“为什么选择这个方案”。

目录示例：

```text
docs/
└── adr/
    ├── 0001-use-postgresql.md
    ├── 0002-use-jwt-authentication.md
    ├── 0003-use-langgraph-for-agent.md
    └── 0004-delay-multi-tenant-support.md
```

单个 ADR 可以采用：

```md
# ADR-0003：使用 LangGraph 管理面试流程

## 状态

已接受

## 背景

面试过程需要维护多轮状态，并支持动态追问。

## 决策

使用 LangGraph 管理面试状态和节点流转。

## 原因

- 状态转换清晰
- 便于插入追问节点
- 支持流程恢复

## 放弃的方案

- 普通 Service 状态机
- 完全由 Prompt 控制流程

## 影响

- Agent 状态必须可序列化
- 业务层不能直接修改流程状态
```

没有 ADR 时，AI 很可能过几轮后推翻原有技术选择。

### 8. `DATA_MODEL.md`

描述数据库模型和数据关系。

建议包含：

* 实体定义
* 字段含义
* 表关系
* 索引设计
* 唯一约束
* 删除策略
* 状态枚举
* 数据生命周期

数据库结构最好仍以迁移文件或 Schema 为最终事实来源，文档主要解释业务含义和设计原因。

### 9. `API.md` 或 OpenAPI 文件

API 最好使用结构化规范维护，例如 OpenAPI，而不是仅依赖自然语言。

需要说明：

* 路径
* 请求参数
* 响应结构
* 错误码
* 权限要求
* 分页规则
* 幂等性
* 示例数据

Agent 修改前后端代码时，可以同时检查接口是否一致。

## 四、AI 专用上下文文档

### 10. `AGENT_GUIDE.md`

这是 Vibe Coding 项目中非常重要的一份文档，用于告诉 Agent 应该如何参与该项目。

可以包含：

```md
# Agent 开发规则

## 基本原则

- 优先修改已有代码，不随意重写模块
- 不引入未经批准的新依赖
- 不为未来需求提前抽象
- 单次任务只处理明确范围
- 遇到架构选择时先检查 ADR
- 修改接口时同步更新 API 文档
- 修改数据库时必须创建迁移文件
- 完成任务后运行相关测试

## 技术约束

- Java 21
- Spring Boot 3
- JavaFX
- PostgreSQL
- 不使用 Lombok
- DTO 使用 record
- 时间统一使用 Instant
- ID 使用 UUID

## 输出要求

每次完成任务后说明：
1. 修改了哪些文件
2. 为什么这样修改
3. 执行了哪些测试
4. 仍有哪些风险
5. 是否需要更新文档
```

这份文档应简短、明确、可执行，不要写大量抽象口号。

### 11. `CONVENTIONS.md`

描述项目编码规范，包括：

* 命名规则
* 目录规则
* 错误处理
* 日志规范
* DTO 规范
* 数据库规范
* 前端组件规范
* 状态管理规范
* 注释规范
* 测试命名规范

不要只写“遵循最佳实践”，而要写成可以判断对错的规则。

例如：

```md
- Service 方法不得直接返回 Controller Response 类型
- Repository 查询不得返回 null，使用 Optional
- 日志中禁止打印 Token、密码和简历原文
- 前端组件超过 300 行时应评估拆分，但不得仅因行数机械拆分
```

## 五、质量与验收相关文档

### 12. `TESTING.md`

说明项目如何测试。

建议包含：

* 测试分层
* 单元测试范围
* 集成测试范围
* 端到端测试范围
* 测试数据策略
* Mock 原则
* 测试命令
* 最低验收要求

例如：

```md
## 完成一个后端功能的最低要求

- 核心 Service 包含单元测试
- Controller 包含正常和异常场景测试
- 数据库查询包含集成测试
- 所有已有测试通过
```

### 13. 功能验收文档

可以放在需求文档或任务中，不一定单独建立文件。

推荐使用 Given / When / Then：

```md
场景：上传不支持的文件

Given 用户已经登录
When 用户上传 DOCX 文件
Then 接口返回 400
And 错误码为 UNSUPPORTED_RESUME_FORMAT
And 文件不会被保存
```

明确验收标准后，AI 才不会只完成“看起来能运行”的实现。

---

## 六、运行与维护文档

### 14. `DEPLOYMENT.md`

包含：

* 构建方法
* 部署流程
* 环境区分
* 配置方式
* 数据库迁移
* 回滚步骤
* 健康检查
* 发布检查清单

### 15. `RUNBOOK.md`

用于记录系统出现问题时如何排查。

例如：

* 服务无法启动
* 数据库连接失败
* Agent 请求超时
* 消息重复消费
* 磁盘空间不足
* 数据迁移失败

Agent 也可以根据 Runbook 辅助排查问题，而不是每次从头推理。

### 16. `CHANGELOG.md`

记录版本层面的用户可感知变化：

```md
## 0.3.0

### Added

- 支持根据简历生成候选人画像

### Changed

- 优化面试记录加载速度

### Fixed

- 修复重复提交答案的问题
```

代码提交历史不能完全替代 Changelog，因为提交记录通常过于细碎。

---

## 七、风险、问题和技术债文档

### 17. `RISKS.md`

记录可能影响项目的风险：

```md
| 风险 | 影响 | 概率 | 应对方式 |
|---|---|---|---|
| 大模型输出格式不稳定 | 高 | 中 | JSON Schema 校验与重试 |
| PDF 解析准确率不足 | 中 | 高 | 保留人工修正入口 |
```

### 18. `TECH_DEBT.md`

记录暂时接受但以后需要处理的问题。

```md
### TD-012 面试记录一次性全部加载

原因：
MVP 阶段实现简单。

当前影响：
记录超过 500 条后加载变慢。

计划：
S1 增加分页查询。

禁止行为：
在没有性能数据前重写整个记录模块。
```

最后一项很重要，可以防止 Agent 看到“不够完美”的代码后自行大规模重构。

# 推荐的项目文档结构

```text
project/
├── README.md
└── docs/
    ├── README.md
    ├── PRODUCT.md
    ├── ROADMAP.md
    ├── STATUS.md
    ├── TASKS.md
    ├── CHANGELOG.md
    ├── AGENT_GUIDE.md
    ├── ARCHITECTURE.md
    ├── DATA_MODEL.md
    ├── API.md
    ├── TESTING.md
    ├── DEPLOYMENT.md
    ├── RUNBOOK.md
    ├── RISKS.md
    ├── TECH_DEBT.md
    │
    ├── adr/
    │   ├── 0001-xxx.md
    │   └── 0002-xxx.md
    │
    └── features/
        ├── authentication.md
        ├── resume-analysis.md
        └── ai-interview.md
```

项目根目录只保留 `README.md` 作为文档入口；其余 Markdown 文档统一放在 `docs/` 内，避免项目元数据与详细设计混杂。

# 个人项目的最小文档组合

不需要一开始创建十几份空文档。对于个人开发或课程设计，建议先维护以下 7 份：

| 文档                | 解决的问题       |
| ----------------- | ----------- |
| 根目录 `README.md`          | 项目是什么、怎么运行       |
| `docs/PRODUCT.md`          | 做什么、不做什么         |
| `docs/ROADMAP.md`          | 项目分几个阶段          |
| `docs/TASKS.md`            | 当前具体做什么          |
| `docs/STATUS.md`           | 现在真实做到哪里         |
| `docs/ARCHITECTURE.md`     | 系统如何组织           |
| `docs/AGENT_GUIDE.md`      | Agent 应该如何修改项目  |

接口复杂后增加 OpenAPI，数据库复杂后增加 `DATA_MODEL.md`，重要技术选择出现后增加 ADR。

# 最重要的维护原则

第一，文档必须有明确的事实优先级。例如：

```text
数据库迁移文件 > DATA_MODEL.md
OpenAPI 文件 > API.md
代码实际实现 > STATUS.md 中的过期描述
已确认 ADR > Agent 自行推测的最佳实践
```

第二，每份文档只承担一种职责。不要把需求、进度、技术设计和开发日志全部塞进一个几万字的文档。

第三，任务完成时同步更新文档，而不是项目结束后补文档。一次 Vibe Coding 任务的完整闭环应当是：

```text
读取项目规则
→ 读取当前任务
→ 检查相关架构决策
→ 修改代码
→ 执行测试
→ 更新任务状态
→ 更新项目状态
→ 必要时新增 ADR 或技术债
```

第四，尽量记录“为什么”，而不只是记录“是什么”。代码通常已经表达了系统做了什么，真正容易丢失的是当初为何这样决定。

第五，文档中应避免“后续优化”“使用最佳实践”“完善异常处理”这类不可验证描述。应改成具体任务、条件和验收标准。
